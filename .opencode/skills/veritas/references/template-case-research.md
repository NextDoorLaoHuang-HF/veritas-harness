# 检索报告参考模板

> 本模板提炼自"租赁合同违约金争议"样板

## 报告类型标识

`--type case-research`

## 元数据头格式

```markdown
# [检索问题标题]

> 来源：[来源名称] · [板块名]
> 作者：[姓名] | 日期：YYYY-MM-DD
> 文档链接：[URL]

---
```

## 正文结构

```markdown
## 一、检索问题

[一句话描述检索问题]

## 二、初步结论

1. [结论1，含具体数据：如"支持了5个月租金的违约金"]

2. [结论2，含对比性判断]

3. [结论3，含综合因素说明]

## 三、研究依据

### （一）参考案例

| 序号 | 概要 | 具体内容 | 依据（案号） |
|------|------|---------|------------|
| 1. | 🔴 [一句话概要，含关键数据] | [法院核心认定，2-4句] | 案例一 |
| 2. | 🔴 [概要] | [内容] | 案例二 |
| ... | ... | ... | ... |

---

### 案例一：[当事人与案由]

**案号**：[案号，如 (2024)京02民终3691号]

[原告诉讼请求和事实理由，保留原文关键句]

[法院认定和判决，**加粗**关键法律推理段落]

[判决主文]

---

### 案例二：[当事人与案由]

**案号**：[案号]

[同上结构]

---

### 案例N：[当事人与案由]

**案号**：[案号]

[同上结构]
```

## 质量标准

| # | 标准 | 说明 |
|---|------|------|
| 1 | **三段式** | 检索问题 → 初步结论 → 研究依据，严格递进 |
| 2 | **结论先行** | 初步结论必须在案例详情之前，每条含具体数据 |
| 3 | **案例概要表** | 必须有四列（序号/概要/具体内容/案号），概要用🔴标记重要程度 |
| 4 | **案号完整** | 每个案例必须有完整案号（如 (2024)京02民终3691号） |
| 5 | **关键推理加粗** | 法院的核心法律推理段落必须 **加粗** |
| 6 | **金额精确** | 判决金额、诉讼请求金额须精确到元/分 |
| 7 | **原文保留** | 法院说理部分保留原文表述，不意译 |
| 8 | **对比提炼** | 初步结论须做跨案例对比，提炼规律性判断 |
| 9 | **案例间分隔** | 案例之间用 `---` 分隔 |

## 必需分节

- 一、检索问题（1句话）
- 二、初步结论（至少2条编号结论，每条含具体数据）
- 三、研究依据
  - （一）参考案例（四列表格）
  - 每个案例单独小节（案号 + 原文 + 加粗重点）

## 证据分配点（反附录式 — 重要）

**特别说明**：case-research 类型的核心载体就是案例本身，**"案例速查表" + "案例详情小节"是结构主体而非附录**。但仍须遵守：

1. **初步结论必须引用案号**——每条结论后注明依据的案号（如"参见(2024)京02民终3691号"），不能脱离案例空谈结论
2. **案例速查表必须按争点/裁判倾向排序**，而非按案号字母序或时间序
3. **案例详情小节中可对比性引用**——如"与案例一相反，案例三法院认为..."，让案号在论理段而非仅在标题出现
4. 案号在论理段中以**加粗**或行内引用形式出现

**反例**（被 verify 标记为"证据未融入论理"）：
- 初步结论 3 条，0 个案号引用
- 案例详情仅"案情+判决"两段，案号间无对比

**正例**：
- 初步结论 3 条，每条挂 1-2 个案号
- 案例三小节中"与案例一相反，本院认为..." 让案号成为论理锚点

## 案例概要表的🔴标记规范

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| 🔴 | 核心案例/高参考价值 | 判决金额明确、裁判规则清晰、与本检索问题高度相关 |
| 🟡 | 参考案例/中等参考价值 | 部分要素可参考但事实差异较大 |
| 🟢 | 辅助案例/低参考价值 | 仅个别要点可参考 |

> 当前样板均使用🔴，实际应按参考价值分级。

## finalize 验证规则

- 检测三段式结构（一、检索问题 / 二、初步结论 / 三、研究依据）
- 检测案例概要表（4列：序号/概要/具体内容/依据）
- 检测每个案例的案号（正则：`\（\d{4}\）.+号` 或 `\d{4}\..+号`）
- 检测加粗关键推理（`**...**` 在案例详情中的数量）
- 检测案例间分隔线（`---`）

## 证据引用方式

检索报告的证据来源主要是裁判文书，不使用通用管线的 S# 标签体系。替代方案：
- 案例概要表的"依据（案号）"列作为引用锚点
- 案例详情的案号作为唯一标识
- 如有法条引用，用法条号（如"《民法典》第五百八十五条"）而非 S#

## 与通用管线的差异

检索报告走**类型验证管线**：
```
veritas search --backend yuandian --type case "..."
→ Agent 写草稿（按本模板）
→ veritas verify --run-dir <path> --type case-research
→ veritas legal verify-citations --run-dir <path> （如含法条引用）
→ veritas finalize --run-dir <path> --type case-research
```

与 general/deep-research 不同的步骤：
- ❌ evidence claim check（案例来自元典API，非网页抓取）
- ❌ finalize 的 S# 标签注入（代码会跳过 S# 标签处理，用法条号和案号替代）
- ❌ 置信度独立章节（置信度嵌入初步结论的判断中）

## 置信度声明

在初步结论最后一条或研究依据末尾，以一段话声明：
- 检索范围（哪些法院、哪些时间段）
- 判例数量与代表性
- 可能的遗漏方向
