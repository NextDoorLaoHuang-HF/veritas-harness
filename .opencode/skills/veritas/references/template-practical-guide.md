# 实务指引参考模板

> 本模板提炼自"知识产权侵权全流程法律要点"样板

## 报告类型标识

`--type practical-guide`

## 元数据头格式

```markdown
# [主题]：[副标题 — 从XX到XX]

> 来源：[来源名称] · [板块名]（[来源类型，如"外部参考收录"]）
> 原文链接：[URL]（如为外部收录）
> 收录时间：YYYY-MM-DD
> 文档链接：[URL]

---

[1-3句痛点引入段落，面向读者的问题意识]

---
```

## 正文结构

```markdown
## 一、[核心前提/基础概念章节]

[2-4句说明当前认知误区或行业现状]

1. **[早期/旧规则]**：[一句话说明旧裁判规则或通行做法]
2. **[现行/新规则]**：[一句话说明当前主流规则，含关键时间节点]
  
[1-2句总结判断，**加粗**核心结论]

---

## 二、[认定标准/核心要件章节]

[1句引入语，说明认定须满足的条件]

1. **[要件一标题]**：[说明，含法律依据]；
2. **[要件二标题]**：[说明，含实务关键证据]；
3. **[要件三标题]**：[说明，含判断标准]。

---

## 三、[法律后果/责任承担章节]

[1句引入语，说明实务关注点]

1. **[责任类型一]**：[说明，含**关键数据**如判赔金额区间]
2. **[责任类型二]**：[说明，**划重点！**标注特别注意事项]
3. **[责任类型三]**：[说明]
4. **[责任类型四]**：[说明]

---

## 四、[应对方案/操作步骤章节]

[1句引入语，给出总体策略方向]

### 1. [第一步：紧急处置]

[具体操作指引，含线上线下排查清单]

### 2. [第二步：双线准备]

- **和解层面**：[说明，含参考价格/协商策略]
- **应诉层面**：[说明，含时限要求]

### 3. [第三步：程序与实体抗辩]

- **程序上**：[说明，含法定时限]
- **实体上**：[说明，含核心抗辩要点]

### 4. [第四步：追责转嫁]

[说明，含合同审查要点和追偿路径]

---

## [总结提示章节标题]

[1-2句核心提示，**加粗**最关键建议]

---

*本文档为[外部资料收录/原创]，仅供内部学习参考*
```

## 质量标准

| # | 标准 | 说明 |
|---|------|------|
| 1 | **中文编号** | 主章节用 一、二、三、四；步骤用 1. 2. 3. 4. |
| 2 | **递进逻辑** | 前提 → 标准 → 后果 → 应对，严格递进 |
| 3 | **痛点引入** | 开头必须面向读者痛点，不用抽象概述 |
| 4 | **金额/阈值** | 涉及赔偿/罚款/时限必须给具体数字，不用"可能""较高"等模糊词 |
| 5 | **划重点** | 实务关键处用 **划重点！** 标注 |
| 6 | **双线方案** | 和解与应诉必须并列给出，不偏废 |
| 7 | **可操作性** | 每一步必须可直接执行，不含"视情况而定"类废话 |
| 8 | **尾注标识** | 文末注明"外部资料收录"或"原创" |

## 必需分节

- 一、基础概念/前提（至少1个编号要点）
- 二、认定标准/核心要件（至少3个要件）
- 三、法律后果/责任承担（至少2项责任）
- 四、应对方案（至少3个步骤，每步可操作）
- 总结提示（1-2句核心建议）

## 证据分配点（反附录式 — 硬性要求）

**核心禁令**：禁止将所有案号/案例堆在文末"附录：参考案例"中。**所有案号必须按争点融入对应章节正文**。

模板骨架的每个主章节内**预留证据占位符**：

```markdown
## 一、[核心前提/基础概念章节]

[正文]

> **📌 证据分配点 1 — 主题：[例如"劳动关系是否成立"]**
>
> 推荐插入 2-3 个相关案例，含完整案号 + 一句话核心规则：
> - `（YYYY）XX民XX号`：[当事人]诉[当事人][案由]案 — [一句话裁判规则]
> - `（YYYY）XX民XX号`：[当事人]诉[当事人][案由]案 — [一句话裁判规则]
>
> 案例来源：`veritas inject-cases --run-dir <path> --type practical-guide` 自动推荐

---

## 二、[认定标准/核心要件章节]
...（同上结构）
```

**占位符执行规则**：

1. 写草稿时，**先填正文再填案例**，不要先收集案例再写
2. 每个主章节的"证据分配点"必须填入 ≥2 个相关案号；无相关案例时**省略该占位符而非留空**
3. 案号在正文中以加粗或行内引用形式出现，如：**参见(2018)湘09行终70号案**
4. `veritas inject-cases --run-dir <path>` 会从 query-*.json / scrape-*.json / scrape-manifest.tsv 自动抽取 case 并按主题词推荐到对应章节，但**最终筛选和插入由 Agent 完成**，CLI 仅供建议
5. **禁用**文末"附录A：参考案例（按案号排序）"等聚合表；如需集中展示，仅保留"案例速查表"且每个案例后注明其在正文中出现的章节锚点（如`§3.2`）

**反例**（被 verify 标记为"证据分布不均"）：
- 全文 20 个案号，19 个在文末附录 A，正文 0 个
- 6 个主章节，5 个章节案号 0 个，仅"法律后果"章节塞 18 个

**正例**：
- 6 个主章节，每章 2-4 个案号，附录"案例速查表"仅 4 列（案号/当事人/争点/正文位置）

## finalize 验证规则

- 检测中文编号章节（一、二、三、）
- 检测步骤编号（### 1. 或 1. 2. 3.）
- 检测至少1处 **划重点** 或等价强调标记
- 检测尾注行（*本文档为...*）

## 置信度格式

实务指引的置信度嵌入正文而非独立章节：
- 法条引用须标注具体条文号
- 金额区间须注明"综合各地生效判例"或具体来源
- 如为外部收录，元数据头须有原文链接

## 与通用管线的差异

实务指引走**类型验证管线**，但与 general/deep-research 不同：
- 内容多来自法律条文和行业经验，证据锚点以法条号和案号为主
- `verify --type practical-guide` 检查中文编号、核心章节、划重点、法条引用精度和质量阈值
- `finalize --type practical-guide` 使用类型验证结果判定 FINAL_STATUS，但跳过 S# 标签注入和证据表改写
- 如含法条引用，仍应额外运行 `legal verify-citations`

推荐的管线：
```
Agent 写草稿（按本模板）
→ veritas verify --run-dir <path> --type practical-guide
→ veritas legal verify-citations --run-dir <path>
→ veritas finalize --run-dir <path> --type practical-guide
```
